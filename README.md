# ✈️ Travel Agent AI

An AI-powered travel itinerary generator built with React, Express, and Mistral AI. Uses a LangGraph multi-step chain to resolve destinations, fetch or generate (fun project, accuracy of the data is not a concern here) travel data, and stream personalized itineraries.

## Setup & Installation

### Prerequisites

- Node.js 18+
- npm 8+
- Mistral AI API key
- Supabase project (postgresql)

### Install

```bash
# Install all dependencies (root + client + server)
npm run install:all
```

### Environment Variables

Create a `.env` file in the `server/` directory:

```env
MISTRAL_API_KEY=your_mistral_api_key_here
SUPABASE_URL=your_supabase_url_here
SUPABASE_KEY=your_supabase_anon_key_here
```

## Development

### Model Configuration

Open `server/src/lib/model.ts` and set `IS_TESTING`:

- `IS_TESTING = true` — uses **Ollama** (`llama3.1`) for local development (no API key needed)
- `IS_TESTING = false` — uses **Mistral AI** (`mistral-small-latest`) for production

To swap in a different model provider, replace `new ChatMistralAI(...)` with any LangChain-compatible chat model. See the full list at https://docs.langchain.com/oss/javascript/integrations/providers/all_providers.

```bash
# Run client and server concurrently
npm run dev
```

- Frontend: http://localhost:5173
- Backend: http://localhost:3001

## API Endpoints

### `POST /api/travel`

Streams a personalized travel itinerary using Server-Sent Events (SSE). Session memory is scoped per `x-session-id` header, enabling follow-up queries like "make it shorter" or "what about winter instead?".

**Headers**

| Header         | Description                                                    |
| -------------- | -------------------------------------------------------------- |
| `x-session-id` | Session ID for conversation memory (falls back to `"default"`) |

**Request Body**

```json
{
  "message": "I want to travel to Japan in spring"
}
```

**SSE Event Types**

| Type     | Description                   |
| -------- | ----------------------------- |
| `status` | Loading status message        |
| `chunk`  | Streamed itinerary text chunk |
| `done`   | Stream complete               |
| `error`  | Error message                 |

**Example**

```bash
curl -X POST http://localhost:3001/api/travel \
  -H "Content-Type: application/json" \
  -H "x-session-id: abc-123" \
  -d '{"message": "winter trip to Japan"}'
```

---

### `GET /internal/data`

Fetches travel data for a destination from Supabase. Used internally by the chain.

**Request Body**

```json
{ "destination": "Bali" }
```

Returns a `TravelDataResult` object with weather, popular spots, foods, hotels, festivals, and travel tips. Returns `404` if the destination is not found.

---

### `POST /internal/data`

Persists AI-generated travel data to Supabase. Called automatically by the chain when no data exists for a destination.

**Request Body**: A `TravelDataResult` object (generated by the AI data tool).

## AI Chain

The backend uses a **LangGraph `StateGraph`** to orchestrate the following pipeline:

```
User query
  → resolve      (resolveQueryTool: picks destination or asks clarifying question)
  → search       (searchTravelTool: fetches data from Supabase)
  → generate     (generateDataTool: AI-generates data if Supabase has none)
  → itinerary    (streams the final itinerary via SSE)
```

### Graph Logic

- After `resolve`: if `needsMoreInfo` is true, the chain ends early and streams back a clarifying question. Otherwise it proceeds to `search`.
- After `search`: if Supabase returns data, skip straight to `itinerary`. If not, go to `generate` first.
- `generate` saves AI-produced data back to Supabase for future requests before passing it to `itinerary`.

### Tools

| Tool               | File                      | Description                                                    |
| ------------------ | ------------------------- | -------------------------------------------------------------- |
| `resolveQueryTool` | `tools/resolver.tool.ts`  | Extracts destination from user query or requests clarification |
| `searchTravelTool` | `tools/retrieval.tool.ts` | Fetches destination data from Supabase via `/internal/data`    |
| `generateDataTool` | `tools/retrieval.tool.ts` | AI-generates realistic travel data and persists it to Supabase |

### Schemas

| Schema                 | File                         | Description                                                  |
| ---------------------- | ---------------------------- | ------------------------------------------------------------ |
| `resolverResultSchema` | `schema/resolver.schema.ts`  | Output of the resolver — candidates or clarifying question   |
| `travelDataSchema`     | `schema/retrieval.schema.ts` | Travel data shape — weather, spots, foods, hotels, festivals |

## Testing

Integration tests are colocated with their tools:

```bash
# Run all tests
npm run test --workspace=server
```

| Test file                     | What it covers                                              |
| ----------------------------- | ----------------------------------------------------------- |
| `tools/resolver.test.ts`      | Resolver returns correct destination or clarifying question |
| `tools/retrieval.test.ts`     | Data generation produces valid `TravelDataResult`           |
| `chains/travel.chain.test.ts` | End-to-end chain with mock data                             |

> **Note:** Tests use `IS_TESTING = true` (Ollama) by default. Make sure Ollama is running locally with `llama3.1` pulled before running tests.

- Download ollama: https://ollama.com/ (More guides and docs available here)
- Open terminal and run the following command
- Pull llama3.1: `ollama pull llama3.1`
- Serve ollama: `ollama serve`
